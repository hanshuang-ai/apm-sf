# SDD 全流程编排（tc-flow）

把一个功能按规范驱动开发的四个核心阶段串起来：`/tc.specify` → `/tc.plan` → `/tc.tasks` → `/tc.implement`。每个阶段结束后停下，等待人工确认，确认后再进入下一阶段。

这个技能本身不实现任何阶段逻辑，只负责编排与确认。各阶段的真正工作由对应的核心命令完成，阶段间状态通过磁盘文件（`spec.md` / `plan.md` / `tasks.md`）传递，不依赖会话记忆。

## 适用与不适用

适用：单个功能的端到端开发，需要在每个产物上做人工把关。

不适用：
- 只想跑其中某一两步——直接调对应的 `/tc.specify` 等命令即可。
- 修 bug——走 `tc-bugfix-*` 系列。
- 既有代码摸底/迁移——走 `tc-brownfield-*` 系列。

## 阶段顺序

| 阶段 | 原命令 | 增强替代 | 产物 |
|------|--------|---------|------|
| 1 规格 | `/tc.specify` | `/tc-brainstorm` | `specs/<feature>/spec.md` |
| 2 计划 | `/tc.plan` + `/tc.tasks` | `/tc-writing-plans` | `specs/<feature>/plan.md`（合并 plan+tasks） |
| 3 实现 | `/tc.implement` | `/tc.implement` | 代码 + 测试 |

**增强替代说明**：
- `/tc-brainstorm`：深度脑暴规格生成（含逐题澄清、2-3 方案权衡、可视化辅助、逐节审批）。复杂功能优先走此路径。
- `/tc-writing-plans`：深度计划生成（含文件结构映射、TDD 步骤模式、无占位符规则）。产出合并的 plan.md（整合计划 + 任务步骤），替代分开的 `/tc.plan` + `/tc.tasks`。

## 开始前：探测断点

编排不假设总是从头开始。先确定当前功能目录，再按产物存在情况决定入口：

1. 找到当前功能目录 `specs/<feature>/`（无法确定时，把用户输入当作新功能的需求描述，从阶段 1 开始）。
2. 按顺序检查产物是否已存在：`spec.md` → `plan.md`。
3. 从第一个缺失产物对应的阶段进入；若全部存在，进入阶段 3 实现。
4. 把判断结果先告诉用户：从哪个阶段接着跑、为什么。确认后再执行。

这样会话中断后再次调用本技能，能从断点续跑，而不是重头来过。

## 阶段执行规则

对每个待执行阶段，依次做三件事：

1. **执行**：调用该阶段的核心命令。用户传入的需求描述作为参数透传给阶段 1 命令；其余阶段命令一般无需额外参数。每个阶段开始前，询问用户选择原命令还是增强替代：
   - 阶段 1："`/tc.specify`（标准规格）还是 `/tc-brainstorm`（深度脑暴规格）？"
   - 阶段 2："`/tc.plan` + `/tc.tasks`（标准计划+任务）还是 `/tc-writing-plans`（深度计划，合并 plan+tasks）？"
2. **报告**：阶段命令结束后，用一两句话说明产出了什么、落在哪个文件。
3. **确认门**：停止输出，明确等待用户回复，不得自行推进到下一阶段。

## 确认门协议

每个阶段结束后，输出形如：

```
阶段 N（<阶段名>）已完成，产物：<文件路径>。
请审阅后回复：
- 继续 —— 进入下一阶段
- 修改 <说明> —— 按说明重跑本阶段
- 中止 —— 停止编排
```

然后停下，等待用户回复：

- 回复"继续"（或同义表达）：进入下一阶段。
- 回复"修改 …"：在本阶段重新执行（带上用户的修改意见），完成后再次停在本阶段的确认门。
- 回复"中止"（或"停"）：停止编排，简要说明停在了哪个阶段、已有哪些产物，便于以后续跑。

进入阶段 3 实现前同样要过确认门。实现阶段命令内部已有自己的 checklist 校验门，本技能不干预其内部逻辑。

## 完成

阶段 3 实现结束后，给出收尾报告：完成的功能、关键产物、测试情况、建议的下一步（如运行 `tc-review` 评审）。仓库根 `PROGRESS.md` 由 implement 完成后的 after_implement 钩子自动更新，无需在此处理。不在确认门处停留。

## 红线

绝不：
- 在任一确认门处不等回复就自行推进到下一阶段。
- 跳过阶段顺序（如未出 `plan.md` 就直接 `/tc.implement`）。
- 替核心命令重新实现阶段逻辑——只调用，不取代。
- 在用户回复"中止"后继续执行。
